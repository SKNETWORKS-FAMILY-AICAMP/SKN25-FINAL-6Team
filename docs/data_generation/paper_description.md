### 관련 논문/레퍼런스

- Self-Instruct (Synthetic Instruction Generation)
    
    https://arxiv.org/abs/2212.10560
    
    **논문 주제:**
    
    사람이 직접 만든 instruction 데이터 없이도, LLM이 스스로 instruction·input·output 데이터를 생성하고 이를 다시 학습에 사용해 instruction-following 능력을 향상시키는 방법을 제안한 논문입니다.
    
    ---
    
    ## 1. 문제의식
    
    기존 Instruction Tuning은 대부분 사람이 작성한 instruction 데이터에 의존합니다.
    
    | 기존 방식의 한계 | 내용 |
    | --- | --- |
    | 데이터 구축 비용 | 사람이 직접 instruction과 정답을 작성해야 함 |
    | 다양성 부족 | 사람이 만든 태스크는 기존 NLP 태스크에 편중됨 |
    | 확장성 한계 | 새로운 사용자형 작업을 대규모로 만들기 어려움 |
    
    즉, **“좋은 LLM을 만들려면 좋은 instruction 데이터가 필요한데, 그 데이터를 사람이 계속 만들기는 어렵다”**는 문제에서 출발합니다.
    
    ---
    
    ## 2. 제안 방법: SELF-INSTRUCT
    
    SELF-INSTRUCT는 LLM이 자기 자신을 활용해 학습 데이터를 만드는 방식입니다.
    
    ### 전체 흐름
    
    1. **Seed Task 준비**
        - 사람이 작성한 소량의 seed task 사용
        - 논문에서는 **175개 seed task**로 시작
    2. **Instruction 생성**
        - 기존 task를 예시로 주고, LLM이 새로운 instruction 생성
    3. **Classification 여부 판단**
        - 생성된 instruction이 분류 문제인지 아닌지 판단
        - 분류 문제와 비분류 문제는 instance 생성 방식이 다름
    4. **Input-Output Instance 생성**
        - 비분류 task: input을 먼저 만들고 output 생성
        - 분류 task: class label을 먼저 만들고 그에 맞는 input 생성
    5. **Filtering**
        - 너무 유사한 instruction 제거
        - 품질 낮은 데이터 제거
        - 이미지·그래프처럼 텍스트 LLM이 처리하기 어려운 task 제거
    6. **Instruction Tuning**
        - 생성된 instruction/input/output 데이터를 사용해 원래 LLM을 fine-tuning
    
    ---
    
    ## 3. 생성된 데이터 규모
    
    논문에서는 GPT-3를 활용해 다음 규모의 데이터를 생성했습니다.
    
    | 항목 | 수치 |
    | --- | --- |
    | 생성 instruction 수 | 52,445개 |
    | classification instruction | 11,584개 |
    | non-classification instruction | 40,861개 |
    | 전체 instance 수 | 82,439개 |
    | empty input instance | 35,878개 |
    
    핵심은 **소량의 seed task만으로 5만 개 이상의 instruction 데이터를 자동 생성했다는 점**입니다.
    
    ---
    
    ## 4. 품질 평가 결과
    
    생성 데이터 품질은 완벽하지 않습니다.
    
    | 평가 항목 | Yes 비율 |
    | --- | --- |
    | instruction이 유효한 task인가 | 92% |
    | input이 instruction에 적절한가 | 79% |
    | output이 정답으로 적절한가 | 58% |
    | instruction/input/output 모두 유효한가 | 54% |
    
    즉, **instruction 자체는 대부분 쓸 만하지만, output 정답 품질은 노이즈가 꽤 있음**을 보여줍니다.
    
    ---
    
    ## 5. 실험 결과
    
    SELF-INSTRUCT로 학습한 GPT-3는 기존 GPT-3보다 instruction-following 성능이 크게 향상되었습니다.
    
    | 모델 | SUPERNI ROUGE-L |
    | --- | --- |
    | Vanilla GPT-3 | 6.8 |
    | GPT-3 + T0 Training | 37.9 |
    | GPT3 Self-Instruct | 39.9 |
    | InstructGPT001 | 40.8 |
    
    핵심 결과:
    
    - SELF-INSTRUCT는 vanilla GPT-3 대비 **+33.1%p 성능 향상**
    - 공개 instruction 데이터로 학습한 모델보다 좋은 성능
    - InstructGPT001과 거의 유사한 수준까지 접근
    - SUPERNI 데이터와 함께 쓰면 추가 성능 향상 가능
    
    ---
    
    ## 6. 프로젝트에 필요한 인사이트
    
    게임 문의 응대 프로젝트 기준으로 필요한 핵심은 이 부분입니다.
    
    ### 1) 합성 데이터 생성 가능성
    
    이 논문은 **실제 고객 문의 데이터가 부족할 때 LLM으로 instruction 데이터를 확장할 수 있다**는 근거가 됩니다.
    
    예시 구조:
    
    | 생성 대상 | 게임 CS 적용 예시 |
    | --- | --- |
    | Instruction | “유저의 결제 누락 문의를 분류하라” |
    | Input | “어제 결제했는데 원석이 안 들어왔어요” |
    | Output | category: 결제, risk_level: MEDIUM, route: rag_reply |
    
    ---
    
    ### 2) 단순 생성만 하면 안 됨
    
    논문에서도 output 정답 품질은 58% 수준입니다.
    
    따라서 프로젝트에 적용할 때는 **생성 → 필터링 → 검수 → 학습/평가 데이터 반영** 구조가 필요합니다.
    
    권장 파이프라인:
    
    ```mermaid
    flowchart TD
        A[Seed 문의 데이터] --> B[LLM 기반 문의/라벨 생성]
        B --> C[중복 제거]
        C --> D[정책·카테고리 규칙 필터링]
        D --> E[샘플 운영자 검수]
        E --> F[학습/평가 데이터셋 반영]
    ```
    
    ---
    
    ### 3) 분류 태스크와 생성 태스크를 나눠야 함
    
    논문은 classification task와 non-classification task를 다르게 생성합니다.
    
    게임 CS에서도 이 구분이 중요합니다.
    
    | 태스크 유형 | 예시 | 생성 방식 |
    | --- | --- | --- |
    | 분류 | 문의 유형, 감성, 위험도, 라우팅 | label 먼저 정의 후 input 생성 |
    | 생성 | 답변 초안, 요약, 운영 인사이트 | input 먼저 생성 후 output 생성 |
    
    ---
    
    ### 4) Seed Task 품질이 중요함
    
    SELF-INSTRUCT는 seed task를 기반으로 확장됩니다.
    
    따라서 seed가 부실하면 생성 데이터도 부실해집니다.
    
    게임 CS에서는 seed task를 아래처럼 구성하는 것이 좋습니다.
    
    | Seed 구성 | 예시 |
    | --- | --- |
    | 문의 유형 | 결제, 환불, 아이템 미지급, 계정, 버그, 이벤트 |
    | 위험 수준 | LOW, MEDIUM, HIGH |
    | 라우팅 | auto_reply, rag_reply, human_review, urgent_alert |
    | 답변 스타일 | 공손함, 근거 포함, 운영자 검수 필요 여부 |
    | 금지 케이스 | 환불 확정 단정, 보상 지급 약속, 개인정보 요청 |
    
    ---
    
    ## 7. 발표/PPT용 한 줄 요약
    
    **SELF-INSTRUCT는 소량의 사람이 만든 seed task를 기반으로 LLM이 instruction·input·output 데이터를 스스로 생성하고, 이를 필터링해 다시 학습에 활용함으로써 instruction-following 성능을 높이는 데이터 부트스트래핑 방법이다.**
    
    프로젝트 적용형으로 쓰면:
    
    **본 프로젝트에서는 SELF-INSTRUCT 방식처럼 실제 문의 데이터가 부족한 초기 단계에서 LLM 기반 합성 문의·라벨 데이터를 생성하되, 중복 제거와 정책 필터링, 운영자 검수를 거쳐 분류·라우팅·답변 생성 성능을 보완할 수 있다.**
    
- Persona 기반 Synthetic User Simulation 연구
    
    https://dl.acm.org/doi/full/10.1145/3708319.3733685
    
    ## 논문 핵심 요약
    
    **Simulating Human Opinions with Large Language Models**
    
    이 논문은 **LLM이 사람의 설문 응답을 얼마나 잘 모방할 수 있는지**를 검증한 연구입니다. 특히 실제 사람의 인구통계 정보를 기반으로 “디지털 트윈”을 만들고, LLM이 그 사람처럼 설문에 응답하게 하는 방식입니다.
    
    ---
    
    ## 1. 연구 목적
    
    기존 설문조사는 비용과 시간이 많이 들고, 응답률 저하로 데이터 품질 문제가 발생합니다.
    
    그래서 이 논문은 다음 질문을 검증합니다.
    
    > **LLM으로 실제 사람의 의견 데이터를 대체하거나 보완할 수 있는가?**
    > 
    
    연구 대상은 정치·의료 같은 민감 영역이 아니라, **탄산음료 브랜드 선호도**라는 일반 소비자 조사입니다.
    
    ---
    
    ## 2. 실험 방식
    
    ### 실제 설문 데이터
    
    | 항목 | 내용 |
    | --- | --- |
    | 대상 | 미국 응답자 |
    | 최종 표본 수 | 461명 |
    | 조사 주제 | 탄산음료 브랜드 인지도·구매·선호 |
    | 브랜드 | Coca-Cola, Pepsi, Sprite, 7UP, Dry, Moxie, Blue Sky, Orangina |
    | 문항 유형 | 이진 선택 문항 + 1~7점 Likert 척도 |
    
    ---
    
    ## 3. ASPIRE 방식
    
    논문은 **ASPIRE**라는 도구를 만들었습니다.
    
    **ASPIRE = Automated Synthetic Persona Interview and Response Engine**
    
    역할은 다음과 같습니다.
    
    1. 실제 응답자의 인구통계 정보 수집
    2. 같은 속성을 가진 synthetic persona 생성
    3. GPT-4o에게 해당 persona 역할 부여
    4. 설문 문항을 순차적으로 제시
    5. 실제 사람 응답과 LLM 응답 비교
    
    사용한 persona 정보는 다음과 같습니다.
    
    Persona 요소
    
    ---
    
    성별
    
    ---
    
    출생연도
    
    ---
    
    교육 수준
    
    ---
    
    직업
    
    ---
    
    재정 상황
    
    ---
    
    거주 주
    
    ---
    
    인종/민족
    
    ---
    
    출생 국가
    
    ---
    
    국적
    
    ---
    
    모국어
    
    ---
    
    즉, 단순히 “미국 소비자처럼 답해줘”가 아니라, **개별 응답자와 동일한 인구통계 프로필을 가진 가상 응답자**를 만든 방식입니다.
    
    ---
    
    ## 4. 주요 결과
    
    ### 1) 전체 경향은 어느 정도 맞춤
    
    LLM은 유명 브랜드와 덜 유명한 브랜드를 구분했습니다.
    
    예를 들어 실제 사람들처럼:
    
    - Coca-Cola, Pepsi 같은 유명 브랜드는 더 잘 알고 있음
    - Moxie, Dry 같은 덜 유명한 브랜드는 덜 알고 있음
    
    즉, **집단 단위의 큰 경향성은 어느 정도 재현 가능**했습니다.
    
    ---
    
    ### 2) 개별 응답 정확도는 우연보다 높음
    
    | 문항 유형 | 실제-합성 응답 일치율 |
    | --- | --- |
    | 이진 선택 문항 | 78.64% |
    | 수치 평점 문항 | 78.65% |
    
    LLM 응답은 완전한 랜덤보다 훨씬 나았지만, 실제 사람의 세부 의견을 정확히 복제하는 수준은 아닙니다.
    
    ---
    
    ### 3) 긍정 응답을 과대평가함
    
    가장 중요한 한계입니다.
    
    LLM은 실제 사람보다 브랜드를 더 긍정적으로 평가하는 경향이 있었습니다.
    
    | 문제 | 설명 |
    | --- | --- |
    | 선택 빈도 과대평가 | 브랜드를 안다고 답하는 비율이 실제보다 높음 |
    | 평점 과대평가 | 브랜드 품질·선호도를 실제보다 높게 평가 |
    | 유명 브랜드 편향 | 잘 알려진 브랜드일수록 긍정 과대평가가 더 큼 |
    
    즉, LLM은 사람보다 **더 무난하고 긍정적인 응답**을 생성하는 경향이 있습니다.
    
    ---
    
    ### 4) 응답 다양성이 부족함
    
    LLM이 만든 synthetic data는 실제 사람 데이터보다 분산이 낮았습니다.
    
    | 실제 사람 | LLM synthetic persona |
    | --- | --- |
    | 의견 차이가 큼 | 응답이 비슷하게 몰림 |
    | 선호가 다양함 | 평균적·무난한 답변이 많음 |
    | 극단적/예외적 응답 존재 | 개별 차이 반영 부족 |
    
    핵심은 **LLM이 평균적 경향은 따라가지만, 사람마다 다른 복잡한 의견 차이는 잘 못 살린다**는 점입니다.
    
    ---
    
    ## 5. 논문 결론
    
    LLM 기반 synthetic survey data는 다음 용도로는 가능성이 있습니다.
    
    | 사용 가능성 높음 | 사용 위험 높음 |
    | --- | --- |
    | 초기 아이디어 검증 | 정밀한 시장조사 대체 |
    | 저위험 컨셉 테스트 | 실제 고객 의사결정 예측 |
    | 대략적인 의견 흐름 파악 | 정책·의료·정치 등 민감 영역 |
    | 설문 설계 사전 점검 | 세부 세그먼트 분석 |
    
    논문의 결론은 명확합니다.
    
    > **LLM synthetic data는 실제 사람 데이터를 완전히 대체하기보다는, 초기 탐색·보조 분석 용도로 제한적으로 활용해야 한다.**

    ## 6. 프로젝트 적용 인사이트
    
    게임 문의 응대 프로젝트에서는 이 논문을 **합성 유저 데이터의 한계 근거**로 쓰기 좋습니다.
    
    ### 적용 가능한 부분
    
    LLM으로 다음 데이터를 만들 수 있습니다.
    
    | 생성 대상 | 예시 |
    | --- | --- |
    | 유저 문의 | “결제했는데 아이템이 안 들어왔어요” |
    | 감성 반응 | 분노, 불만, 중립, 만족 |
    | 후속 반응 | “답변이 이해가 안 됩니다”, “아직 해결 안 됐어요” |
    | 유저 persona | 신규 유저, 과금 유저, 복귀 유저, 고레벨 유저 |
    | 문의 패턴 | 반복 문의, 환불 요구, 버그 신고 |

    | --- | --- |
    | 긍정 응답 과대평가 | 실제 유저 불만 강도를 낮게 만들 수 있음 |
    | 응답 다양성 부족 | 다양한 말투·분노 표현·비정상 케이스 부족 |
    | 평균적 persona 생성 | 고위험 유저, 악성 유저, 반복 민원 유저 재현 부족 |
    | 실제 행동 복잡성 부족 | 재문의율·이탈 가능성 예측에는 부정확 |
    
    따라서 게임 CS 합성 데이터는 **그냥 LLM에게 만들게 하면 안 되고**, 실제 문의 로그 기반 seed, 규칙 기반 필터링, 운영자 검수가 필요합니다.
    
    ---
    
    ## 7. PPT용 요약 문장
    
    **LLM은 실제 사람의 설문 응답을 집단 수준에서는 어느 정도 모방할 수 있지만, 긍정 응답을 과대평가하고 개인별 의견 다양성을 충분히 반영하지 못한다. 따라서 synthetic user data는 실제 데이터를 대체하기보다 초기 실험과 데이터 보강을 위한 보조 수단으로 활용하는 것이 적절하다.**
    
    프로젝트 연결 문장:
    
    **게임 문의 응대 시스템에서도 LLM 기반 합성 문의 데이터는 초기 분류·라우팅 모델 보강에는 활용 가능하지만, 실제 유저 불만 강도와 다양한 문의 패턴을 완전히 대체할 수 없으므로 실제 로그 기반 검증과 운영자 검수가 필요하다.**
    
- Synthetic User Generated Text 평가 연구
    
    https://aclanthology.org/2025.cl-1.6.pdf
    
    ## 논문 핵심 요약
    
    **Evaluating Synthetic Data Generation from User Generated Text**
    
    이 논문은 **사용자 생성 텍스트(User Generated Content, UGC)를 합성 데이터로 만들 때 무엇을 기준으로 평가해야 하는가**를 다룬 논문입니다. 단순히 “합성 데이터로 모델 성능이 오르는가”만 보는 것이 아니라, **의미 보존·스타일 보존·개인정보 보호·다운스트림 성능**을 함께 평가해야 한다고 제안합니다.
    
    ---
    
    ## 1. 문제의식
    
    UGC 데이터는 소셜미디어 글, 리뷰, 대화 로그, 커뮤니티 게시글처럼 실제 사용자가 만든 텍스트입니다.
    
    이 데이터는 실제 행동과 감정을 담고 있어 가치가 크지만, 다음 문제가 있습니다.
    
    | 문제 | 내용 |
    | --- | --- |
    | 개인정보 위험 | 글 안에 위치, 성별, 나이, 직업, 정체성 등이 드러날 수 있음 |
    | 라벨 부족 | 전문가 라벨링 비용이 큼 |
    | 데이터 공유 어려움 | 민감 정보 때문에 외부 공유·API 활용 제한 |
    | 합성 데이터 품질 불확실 | 생성된 데이터가 실제 데이터를 대체할 수 있는지 평가 기준 부족 |
    
    따라서 논문은 **합성 텍스트 데이터를 만들 때 품질과 개인정보 보호를 동시에 평가하는 프레임워크**를 제안합니다.
    
    ---
    
    ## 2. 논문이 제안한 평가 기준
    
    이 논문의 핵심은 합성 UGC를 아래 4가지 축으로 봐야 한다는 점입니다.
    
    | 평가 축 | 의미 | 예시 지표 |
    | --- | --- | --- |
    | **Meaning Preservation** | 원문의 의미가 유지되는가 | BERTScore, Fréchet Distance |
    | **Style Preservation** | 사용자의 말투·문체가 유지되는가 | POS score, style embedding similarity |
    | **Divergence** | 원문과 충분히 달라졌는가 | Self-BLEU, Character JSD |
    | **Privacy** | 사용자 식별·속성 추론 위험이 줄었는가 | author profiling, re-identification accuracy |
    
    중요한 점은 **의미 보존과 개인정보 보호는 충돌할 수 있다**는 것입니다.
    
    원문과 너무 비슷하면 의미와 스타일은 잘 보존되지만 개인정보 위험이 커지고, 너무 다르게 만들면 개인정보는 보호되지만 라벨 유효성과 모델 성능이 떨어질 수 있습니다.
    
    ---
    
    ## 3. 실험한 합성 데이터 생성 방식
    
    논문은 여러 생성 방식을 비교했습니다.
    
    | 방식 | 설명 |
    | --- | --- |
    | STRAP | 스타일을 줄이는 paraphrasing |
    | Back-Translation | 독일어·중국어 등을 거쳐 다시 영어로 번역 |
    | Syntax-guided paraphrase | 문장 구조를 조절해 paraphrase |
    | StyMask | 규칙 기반 스타일 마스킹 |
    | DP-BART | 차등 개인정보보호 기반 rewriting |
    | LLaMA paraphrasing | LLM에게 다양한 paraphrase 생성 요청 |
    
    핵심은 **생성 방식마다 성능·문체 보존·개인정보 보호 수준이 다르므로, 목적에 맞게 선택해야 한다**는 것입니다.
    
    ---
    
    ## 4. 실험 대상 태스크
    
    논문은 UGC를 세 가지 수준으로 나누어 실험했습니다.
    
    | 수준 | 데이터/태스크 | 특징 |
    | --- | --- | --- |
    | Post-level | Twitter, Yelp 감성분류 | 짧은 단일 문장/게시글 |
    | Dialogue-level | SwDA 대화행위 분류 | 대화 맥락 필요 |
    | Timeline-level | TalkLife 감정 변화 탐지 | 사용자별 시간 흐름 필요 |
    
    결과적으로 **단순 게시글 분류는 합성 데이터로도 꽤 잘 되지만, 대화·타임라인처럼 맥락과 개인성이 중요한 태스크에서는 성능 차이가 커졌습니다.**
    
    ---
    
    ## 5. 주요 결과
    
    ### 1) 단순 태스크에서는 합성 데이터도 어느 정도 유용함
    
    감성분류처럼 짧고 명확한 태스크에서는 합성 데이터로 학습한 모델도 실제 데이터 학습 모델과 비슷한 성능을 냈습니다.
    
    하지만 복잡한 태스크로 갈수록 성능 격차가 커졌습니다.
    
    | 태스크 난이도 | 합성 데이터 효과 |
    | --- | --- |
    | 짧은 게시글 감성분류 | 비교적 효과적 |
    | 대화행위 분류 | 일부 성능 저하 |
    | 사용자 타임라인 감정 변화 탐지 | 성능 저하·희귀 클래스 문제 발생 |
    
    ---
    
    ### 2) 의미와 스타일 보존이 성능에 중요함
    
    합성 데이터가 원문의 의미와 스타일을 잘 유지할수록 downstream task 성능이 좋아지는 경향이 있었습니다.
    
    특히 사용자 텍스트에서는 단순 의미뿐 아니라 **말투, 문장 구조, 반복 표현, 구어체성**도 중요합니다.
    
    게임 문의로 치면 다음과 같습니다.
    
    | 보존해야 하는 요소 | 예시 |
    | --- | --- |
    | 의미 | “결제했는데 아이템이 안 들어왔다” |
    | 감정 | 분노, 불만, 혼란 |
    | 스타일 | 짧은 항의문, 장문 설명, 비속어, 반복 문의 |
    | 맥락 | 이전 문의 이후 다시 항의하는 상황 |
    
    즉, 합성 문의 데이터는 **내용만 맞으면 되는 것이 아니라 실제 유저 말투와 불만 표현도 반영해야 함**을 보여줍니다.
    
    ---
    
    ### 3) 개인정보 보호를 강화하면 성능이 떨어질 수 있음
    
    DP-BART처럼 개인정보 보호가 강한 방식은 사용자 식별 위험을 줄였지만, 의미·스타일 보존이 약해져 모델 성능이 떨어지는 경우가 있었습니다.
    
    | 방향 | 장점 | 단점 |
    | --- | --- | --- |
    | 원문과 비슷하게 생성 | 라벨 유지, 성능 좋음 | 개인정보 위험 증가 |
    | 원문과 다르게 생성 | 개인정보 보호 강화 | 의미 손상, 성능 저하 가능 |
    
    따라서 합성 데이터는 **목적에 따라 trade-off를 조정해야 합니다.**
    
    ---
    
    ### 4) 저빈도 클래스에서 편향 증폭 가능
    
    논문은 합성 데이터가 희귀 클래스를 더 약하게 만들 수 있다고 지적합니다.
    
    예를 들어 TalkLife 감정 변화 탐지에서 드문 라벨은 합성 데이터 학습 시 더 과소예측되는 문제가 나타났습니다.
    
    게임 CS에 적용하면 다음 위험이 있습니다.
    
    | 희귀 케이스 | 합성 데이터 위험 |
    | --- | --- |
    | 고위험 민원 | 적게 생성되거나 약하게 표현될 수 있음 |
    | 환불·법적 항의 | 실제보다 단순하게 생성될 수 있음 |
    | 악성 유저/욕설 | 필터링 과정에서 과도하게 제거될 수 있음 |
    | 장애·버그 급증 문의 | 일반 문의처럼 평탄화될 수 있음 |
    
    즉, 합성 데이터를 만들 때 **HIGH risk, urgent_alert, human_review 케이스는 의도적으로 보강해야 합니다.**

## DB 문서 반영 메모

`docs/DB/descriptions.md`, `docs/DB/notion_data.md`, `docs/DB/db_info.md`를 보면 현재 프로젝트는 이미 합성 전제가 아닌 **실제 문의 테이블을 중심으로 동작하는 구조**를 갖고 있습니다.

핵심 사실:

- live DB 기준 `qa_ticket`는 약 9,243건이다.
- `community_users`, `game_accounts`는 각각 약 6,288건이다.
- `qa_ticket`는 `source_type`, `status`, `session_id`, `responder_type`를 포함한다.
- `qa_ticket.account_id`는 nullable이고 `game_accounts`와 연결된다.
- 적재 노트북은 `qa_ticket`의 `account_id`-`user_id` 매핑을 바탕으로 `game_accounts`를 구성한다.
- 현재 스키마에는 별도 챗봇 대화 로그 테이블이 없다.

이 구조는 데이터 생성 전략을 다음처럼 바꾸게 만든다.

### 1) `qa_ticket`는 seed가 아니라 기준축이다

초기 단계에서 실문의가 부족한 상황을 가정한 논문 인사이트는 여전히 유효하지만, 우리 프로젝트는 이미 `qa_ticket`라는 실문의 중심 테이블을 갖고 있다.

따라서 적용 방식은:

- 실문의가 없는 상태에서 전량 합성
- 가 아니라
- **실제 `qa_ticket` 분포를 기준으로 필요한 부분만 제한적으로 보강**

이 되어야 한다.

### 2) 게시글 문의는 실제 분포 기반으로 가야 한다

사용자 요청대로 `qa_ticket` 안에는 실제 게시글/문의 데이터가 존재하므로, 챗봇이 아닌 게시글 문의는 실제로 확인 가능한 seed다.

그래서 `community` 계열 문의는:

- 실제 카테고리 분포
- 실제 길이 분포
- 실제 불만 강도 분포
- 실제 재문의 분포
- 실제 `account_id` null 비율

을 먼저 계산하고, 그 분포를 유지하는 방식으로 정제/보강해야 한다.

즉 `community` 문의는 "새로 쓰기"보다 "정제 후 유지"가 우선이다.

다만 여기서 유지해야 하는 것은 문의 본문과 이용 맥락이지, `source_type`, `status`, `responder_type`, 길이, 재문의 여부 같은 운영 메타값의 분포를 먼저 고정하는 것은 아니다. 이 값들은 생성 후 AI가 판정하거나 규칙 기반으로 부여할 수 있다.

### 3) 챗봇 문의는 간접 재현만 가능하다

현재 스키마에 챗봇 메시지 turn-by-turn 로그가 없기 때문에, `chatbot` 문의는 대화 자체를 완전 복원할 수 없다.

대신 아래 정보만 직접 활용 가능하다.

- `source_type='chatbot'`
- `session_id`
- `responder_type`
- 최종 ticket 텍스트

따라서 챗봇 데이터는:

- 세션 맥락
- bot/human/hybrid 이관 가능성
- 최종 문의 유형

를 기준으로 **ticket 단위에서만** 재현해야 한다.

- hard case와 희귀 라벨을 별도 quota로 증강
- 생성 후 필터링, 정책 검수, 분포 재검증 수행

Synthetic Persona 연구 관점:

- LLM이 불만 강도를 평균화하지 않게 주의
- 반복 민원, 제재 항의, 환불 강경 요구 같은 고위험 패턴을 의도적으로 유지

Synthetic UGC 평가 연구 관점:

- 의미 보존, 스타일 보존, 개인정보 보호, downstream 성능을 함께 평가
- 특히 `community` 실문의 분포를 합성이 과도하게 덮어쓰지 않는지 별도 점검

### 프로젝트 적용 문장

**본 프로젝트는 게임 유저 이용 데이터와 `qa_ticket` 본문을 먼저 만들고, 실제 게시글 문의는 정제 후 최대한 유지하며, 챗봇 문의는 `session_id` 기반으로 간접 재현하고, `source_type`·`status`·`responder_type`·길이·재문의 여부는 생성 후 AI가 판정하는 방식이 적절하다.**
